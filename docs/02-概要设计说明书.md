## 4. 核心数据流

认证: POST /api/public/login -> 识别 userType -> 路由到对应界面
消息: CloakBrowser(CDP) -> Rust收集器 -> 本地SQLite -> 翻译 -> 前端
翻译: 请求 -> 缓存检查 -> 引擎翻译 -> 写缓存 -> 返回
同步: 自定同步联系人摘要(POST /api/client/sync/contacts), 手动全量同步
计费: 选套餐 -> 校验 -> 扣款 -> 激活 (Feature Flag 下次同步生效)

### Svelte 5 选型

Vue 3 -> Svelte 5 理由: 编译产物 2KB(无Runtime), Tailwind 样式控制更精细,
Skeleton UI 组件库颜值更高. 代价是前端资产需从 Vue 重写.

### postgreSQL 替换理由

联系人数据量大，需要跨平台重粉检测（模糊匹配+数组去重），PostgreSQL 的 JSONB + GIN 索引 + 窗口函数 +
CTE 在百万级大表场景远优于 MySQL。

### CloakBrowser 分离下载

安装包体积: ~8MB (不含 CloakBrowser Chromium 引擎).

首次启动流程:
1. 检测 cloak/ 目录是否存在
2. 不存在 -> 显示 '正在准备浏览器环境' + 进度条
3. 从 CDN 下载 CloakBrowser 压缩包 (~150MB)
4. SHA256 校验完整性
5. 解压到 app_data_dir/cloak/
6. 写入版本号 -> 启动完成

后续启动: 检测版本, 如需更新后台静默下载覆盖.

CloakBrowser CDP -> Network.webSocketFrameReceived / Network.responseReceived -> Rust 协议解析器
-> 结构化消息数据 -> 本地 SQLite

注入脚本只做 UI（翻译悬浮窗、快速回复按钮、AI 面板），不再负责数据提取

各平台协议层方案：

WhatsApp - WWebSocket 帧拦截（baileys 协议已被逆向）
Telegram - MTProto WebSocket 拦截
LINE - XHR 响应拦截
Facebook/Messenger - GraphQL XHR 拦截
TikTok/Instagram - XHR + WebSocket 拦截

## 客户端双层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    MetaChat 桌面客户端                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────┐     ┌──────────────────────────────┐  │
│  │  界面层 (UI)     │     │  数据层 (Protocol)    ←核心资产 │  │
│  │                 │     │                              │  │
│  │  Svelte 5 自建界面  │     │  protocol/                    │  │
│  │  (逐步替换)      │     │  ├─ whatsapp.rs             │  │
│  │                 │     │  ├─ telegram.rs              │  │
│  │  + 注入浮层      │     │  ├─ line.rs                 │  │
│  │  (嵌入式浏览时)   │     │  ├─ facebook.rs             │  │
│  └────────┬────────┘     │  └─ mod.rs (Message trait)  │  │
│           │              └─────────────┬────────────────┘  │
│           │  invoke()                  │  CDP              │
│           ▼                            ▼                   │
│  ┌──────────────────────────────────────────────┐         │
│  │         Rust 后端层 (Tauri)                    │         │
│  │                                               │         │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │         │
│  │  │ cloak/   │ │ storage/ │ │ translation/  │ │         │
│  │  │ CDP 池   │ │ SQLite   │ │ 翻译引擎      │ │         │
│  │  │ 实例管理  │ │ 数据模型  │ │ 缓存          │ │         │
│  │  └──────────┘ └──────────┘ └──────────────┘ │         │
│  └──────────────────────────────────────────────┘         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**策略：**

Phase 1: 协议解析器 + 嵌入式浏览（CloakBrowser 加载原生页面，注入 UI 浮层）
Phase 2+: 数据层稳定后逐个平台替换为自建 Vue 界面
- 消息列表 -> 自建
- 发送消息 -> 自建
- 联系人管理 -> 自建
- 最终 CloakBrowser 退化为"认证代理"

**关键设计原则：**

1. protocol/ 模块是独立的核心资产，不依赖 UI，CLI 就能跑
2. 所有平台实现统一的 Message trait: fn send(), fn receive(), fn list_contacts()
3. 嵌入式浏览时：CloakBrowser 全页面加载 + CDP 拦截 + 注入浮层
4. 自建界面时：CloakBrowser 只做登录认证 + CDP 数据通道，不渲染页面
5. 两种模式可以按平台独立切换，不需要一刀切
