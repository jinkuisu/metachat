## 三、核心设计决策

### 3.1 客户端架构 (Tauri 2.0 + Rust + Svelte 5)

| 层面    | 技术                             | 说明                               |
| ----- | ------------------------------ | -------------------------------- |
| 桌面框架  | Tauri 2.0                      | Electron替换, 安装包减少80%             |
| 后端语言  | Rust                           | 异步 (tokio), 管理20+ CloakBrowser实例 |
| UI 框架 | Svelte 5 + Skeleton + Tailwind | 编译产物~2KB, 无Runtime               |
| 浏览器引擎 | CloakBrowser (Chromium)        | 源码级指纹伪装, 独立代理                    |
| 本地存储  | SQLite (rusqlite)              | 消息/联系人/账号/翻译缓存                   |
| 实时通信  | MQTT (rumqttc)                 | 桌面↔手机同步                          |
| 协议截获  | CDP (Chrome DevTools Protocol) | 替代DOM抓取                          |

### 3.2 服务端架构 (Spring Boot 3.x + PostgreSQL)

| 层面  | 技术                          | 说明                  |
| --- | --------------------------- | ------------------- |
| 框架  | Spring Boot 3.4.x + Java 21 |                     |
| ORM | MyBatis-Plus                |                     |
| 数据库 | PostgreSQL 15+              | JSONB + GIN索引, 适合大表 |
| 缓存  | Redis 7.x                   |                     |
| 容器  | Undertow                    | 替代Tomcat, 异步性能更好    |
| 鉴权  | Sa-Token 1.39               | JWT, 双登录类型          |
| 消息  | RabbitMQ/EMQX               |                     |

### 3.3 管理后台

两个独立 SPA，共用统一登录入口，根据 userType 路由到不同前端：

- **Ops Portal**：运维人员管理系统配置、计费、AI、审核
- **User Admin Portal**：销售人员管理客户 + 客户自助服务

### 3.4 手机端 (Phase 4)

通过 MQTT 与桌面端通信，实现实时通知 + 快速回复

## 四、重构阶段划分

### Phase 1: 桌面客户端 (当前)

- Tauri + Rust + Svelte 5 框架搭建
- CloakBrowser 集成 + CDP 协议层
- SQLite 数据迁移引擎
- 核心 UI: 账号管理/会话/翻译/AI/回复
- 后端 API: 先定义接口契约, 本地 mock

### Phase 2: 管理后台 + 后端 API

- Spring Boot 后端重构 (MySQL→PostgreSQL)
- Ops Portal + User Admin Portal
- 对接 Phase 1 定义的接口契约

### Phase 3: 后端 API 全面对接

- 客户端与服务端 API 正式对接
- 订阅/计费/授权体系

### Phase 4: 手机端

- Flutter App
- MQTT 实时同步
- 与客户端联合发布

## 五、关键数据流

### 消息流入

```
社交平台 → CloakBrowser(CDP) → protocol/ 解析
  → 写入本地 SQLite → emit Tauri 事件 "new_message"
  → Svelte 前端更新 unreadCount/消息列表
  → MQTT 推送 (如手机端在线)
```

### 翻译流

```
Svelte invoke("translation_translate")
  → Rust 查缓存 (SQLite) → 未命中 → 调用翻译引擎
  → 写缓存 → 返回结果 → emit 到前端
```

### 认证流

```
登录 → POST /api/public/login → 查sys_user/mc_user
  → BCrypt校验 → 生成JWT → 返回userType + 路由
  → 前端根据角色渲染对应界面
```

## 六、API 路由规划

```
/api/public/*      - 公开 (无认证)
/api/client/*      - 桌面客户端 (JWT mc_user)
/api/ops/*         - 运维端 (JWT sys_user ops)
/api/admin/*       - 用户管理端 (JWT sys_user + mc_user)
/api/internal/*    - 内部服务 (IP白名单)
```
