# MetaChat 重构项目 · 数据迁移详细设计说明书

## 1. 概述

新版本(Tauri)首次启动时，自动检测旧版本(Electron)的 SQLite 数据库并迁移数据。
迁移完成后写入新版本标记，下次不再触发。

## 2. 旧数据库定位

| 平台 | 路径 |
|------|------|
| Windows | %APPDATA%/yituoke/database/ |
| macOS | ~/Library/Application Support/yituoke/database/ |
| Linux | ~/.config/yituoke/database/ |

检测方式: 扫描上述路径是否存在 sqlite3.db 或 database.db

## 3. 旧表结构

旧项目客户端(better-sqlite3)使用的表:

| 旧表名 | 用途 | 字段(重要) |
|--------|------|-----------|
| session | 平台会话 | id, type, url, userId, account, nickname, avatar, settings, sortOrder, online |
| friend | 联系人 | id, platformContactId, name, avatar, remarkName, remarkAge, remarkBirthday, labels, addedAt |
| label | 标签 | id, name, color, accountId |
| friend_label | 联系人-标签关联 | friendId, labelId |
| reply | 快捷回复 | id, groupId, title, content, sortOrder, type |
| reply_group | 回复分组 | id, name, accountId, sortOrder |
| translation_cache | 翻译缓存 | cacheKey, resultJson, expiresAt, lastAccessedAt |
| account_settings | 账号设置 | accountId, settingsJson |
| settings_state | 全局设置 | key, valueJson |
| kv_store | 键值存储 | key, value |
| sync_outbox | 同步队列 | id, type, payload, status, retryCount |
| media_cache | 媒体缓存 | url, path, size |

## 4. 新旧表映射

| 旧表 | 新表 | 迁移策略 |
|------|------|----------|
| session | sessions | 直接映射, settingsJson 解析后存入对应字段 |
| friend | contacts | remark字段展平为 remarks JSON, labels转为标签ID数组 |
| label | labels | 直接映射 |
| friend_label | contacts.labels | 合并到contacts的labels字段 |
| reply + reply_group | reply_rules / reply_templates / reply_groups | 拆分映射 |
| translation_cache | translation_cache | 直接映射 |
| account_settings | account_settings | 格式调整后迁移 |
| settings_state | settings | key直接映射 |
| kv_store | settings | key冲突时settings_state优先 |
| sync_outbox | — | 丢弃(同步策略变更) |
| media_cache | — | 丢弃(由CloakBrowser管理) |

## 5. 迁移引擎设计

```rust
// === storage/migrate.rs ===

pub struct MigrationEngine {
    old_db_path: Option<PathBuf>,
    new_db: rusqlite::Connection,
    report: MigrationReport,
}

impl MigrationEngine {
    /// 检测旧数据库是否存在
    pub fn detect_old_db(&self) -> Option<OldDbInfo>;

    /// 执行全部迁移
    pub fn migrate(&mut self) -> AppResult<MigrationReport>;

    /// 回滚(删除新库, 保留旧库)
    pub fn rollback(&self) -> AppResult<()>;
}

pub struct OldDbInfo {
    pub path: PathBuf,
    pub version: i32,
    pub session_count: u64,
    pub friend_count: u64,
    pub reply_count: u64,
    pub translation_cache_count: u64,
}

pub struct MigrationReport {
    pub success: bool,
    pub migrated_sessions: u64,
    pub migrated_contacts: u64,
    pub migrated_labels: u64,
    pub migrated_replies: u64,
    pub migrated_translation_cache: u64,
    pub migrated_settings: bool,
    pub errors: Vec<String>,
    pub duration_ms: u64,
}
```

## 6. 迁移流程

```
1. 检测旧库是否存在
   ├─ 不存在 → 跳过迁移, 直接启动
   └─ 存在 → 进入迁移流程

2. 读取旧 Schema 版本号
   ├─ 从 kv_store WHERE key = "schema_version" 读取
   └─ 不存在则按最新旧版处理

3. 创建新库 Schema
   ├─ 执行 CREATE TABLE IF NOT EXISTS ... (同03文档)
   └─ 写入 schema_version = 1

4. 逐表迁移(带事务)
   ┌─ session → sessions
   ├─ friend → contacts (含 friend_label 合并)
   ├─ label → labels
   ├─ reply_group + reply → reply_groups + reply_templates
   ├─ translation_cache → translation_cache
   ├─ account_settings → account_settings
   └─ settings_state + kv_store → settings
   每步完成后记录进度, 失败可回滚到上一步

5. 数据完整性校验
   ├─ 行数对比: 旧count == 新count
   ├─ 抽样检查: 随机取5条, 对比关键字段
   └─ 全部通过 → 标记 migration_complete = true

6. 写入完成标记
   INSERT INTO schema_version (version, applied_at, description)
   VALUES (1, UNIXEPOCH(), "Migration from old Electron version")

7. 清理
   └─ 可选: 重命名旧库为 database.old.db 作为备份
```

## 7. 回滚策略

| 场景 | 处理方式 |
|------|----------|
| 单表迁移失败 | 回滚当前事务, 重试3次, 仍失败则跳过该表 |
| 校验失败 | 记录失败详情, 用户可选择继续或回滚全部 |
| 迁移过程崩溃 | 下次启动检测到旧库存在且新库不完整, 删除新库重来 |
| 用户取消 | 停止迁移, 旧库保持不变, 下次启动再次提示 |

## 8. 旧备份兼容

旧版本的备份文件(.db或.json)在新版本中恢复时:
1. 检测备份文件格式版本号
2. 如果是旧格式, 走同样的表映射逻辑进行转换
3. 如果是新格式(同新SQLite schema), 直接覆盖

## 9. 关键SQL转换示例

```sql
-- friend → contacts
INSERT INTO contacts (id, platform, platform_contact_id, account_id,
  name, remark_json, labels_json, added_at, last_active_at)
SELECT
  hex(randomblob(16)),
  "whatsapp",  -- platform从session表获取
  f.platformContactId,
  f.accountId,
  f.name,
  json_object(
    "remarkName", f.remarkName,
    "remarkAge", f.remarkAge
  ),
  f.labels,
  f.addedAt,
  f.lastActiveAt
FROM friend f;

-- reply_group + reply → new tables
INSERT INTO reply_groups (id, name, account_id, sort_order, created_at)
SELECT id, name, accountId, sortOrder, createdAt FROM reply_group;

INSERT INTO reply_templates (id, group_id, title, content, created_at)
SELECT id, groupId, title, content, createdAt FROM reply;
```
