## 1. 项目概述

### 1.1 项目名称

MetaChat（译拓客）重构项目

### 1.2 项目背景

MetaChat 是一个多平台社交消息聚合管理系统，当前基于 Electron + Spring Boot 构建。由于 Electron 体积大、内存占用高、反检测能力不足等问题，决定进行全量重构，优先重构桌面客户端（Electron → Tauri 2.0 + CloakBrowser），后端和管理后台随后分阶段重构。客户端设计阶段需预先定义好与后端交互的 API 接口契约，确保后续后端重构完成后无缝切换。

### 1.3 项目目标

| 目标     | 说明                                              |
| ------ | ----------------------------------------------- |
| 桌面端轻量化 | Electron 迁移到 Tauri 2.0，安装包体积减少 80%+，内存占用降低 50%+ |
| 反检测能力  | 集成 CloakBrowser，源码级指纹伪装，降低社交平台风控风险              |
| 多端实时   | MQTT 实现桌面 ↔ 手机实时消息同步                            |
| 本地优先   | 所有消息/联系人/配置存储在本地 SQLite，云端只做路由转发                |
| 数据兼容   | 新系统安装后自动从旧项目 SQLite 数据库迁移数据                     |
| 商业模式   | 控制台管理用户订阅、设备授权、功能开关                             |
| 自建翻译   | 低成本自建 Nginx 翻译代理，替代高价第三方 API                    |

### 1.4 重构优先级

| 阶段          | 范围               | 说明                         |
| ----------- | ---------------- | -------------------------- |
| **Phase 1** | **桌面客户端（Tauri）** | 独立可运行，后端接口先定义契约，本地 mock 开发 |
| Phase 2     | 管理后台（重构）         | 前后端分离重构                    |
| Phase 3     | 后端 API（重构）       | 对接客户端已定义的接口契约              |
| Phase 4     | 手机端（Flutter）     | MQTT 实时同步，与客户端联合发布         |

## 2. 用户角色定义

| 角色       | 说明                                    | 使用终端                      |
| -------- | ------------------------------------- | ------------------------- |
| **终端用户** | MetaChat 桌面端使用者，管理社交账号、收发消息、翻译、AI 对话等 | Windows/macOS/Linux 桌面客户端 |
| **移动用户** | 通过手机端实时接收消息、回复（未来 Phase 4）            | iOS/Android 手机 App        |
| **管理员**  | 管理后台操作，管理用户、订阅、系统配置、内容审核              | Web 管理后台                  |
| **系统用户** | 拥有后台管理账号的内部运营人员，按 RBAC 权限分配           | Web 管理后台                  |

## 4. 数据需求与迁移

### 4.1 本地数据存储

| 数据类型 | 存储引擎          | 说明                     |
| ---- | ------------- | ---------------------- |
| 会话消息 | SQLite        | 各平台消息内容，按时间索引          |
| 联系人  | SQLite        | 联系人信息、备注字段、标签          |
| 账号配置 | SQLite        | 账号登录态、Cookie、代理配置      |
| 翻译缓存 | SQLite        | 翻译结果缓存，减少重复请求          |
| 用户设置 | SQLite + 配置文件 | 主题、语言、快捷键、性能设置         |
| 备份文件 | 文件系统          | 导出为 SQLite 文件或 JSON 格式 |

### 4.2 旧到新数据迁移

```
┌─────────────────────┐         ┌─────────────────────┐
│    旧 Electron 端    │         │    新 Tauri 端      │
│                     │         │                     │
│  sqlite3.db (消息)   │ ──────→ │  metachat.db (消息)  │
│  config.json (设置)  │ ──────→ │  config.json (设置)  │
│  backups/* (备份)    │ ──────→ │  backups/* (备份)    │
│  userData/* (账号)   │ ──────→ │  profiles/* (账号)   │
└─────────────────────┘         └─────────────────────┘
```

**迁移策略**：

1. 新版本首次启动时自动检测旧项目数据目录
2. 读取旧 SQLite 数据库中的 Schema 版本号
3. 执行 SQL 升级脚本，将旧 Schema 转换为新 Schema
4. 逐表迁移数据，每次迁移完成后记录迁移状态
5. 迁移完成后清理临时文件，写入新版本标记
6. 若迁移失败，自动回滚并提示用户联系支持

### 4.4 数据同步策略

#### 4.4.1 原则

| 原则       | 说明                             |
| -------- | ------------------------------ |
| **本地优先** | 所有消息、联系人、账号配置首先存储在用户本地 SQLite  |
| **按需同步** | 默认仅同步必要的元数据到服务端，非全量同步          |
| **手动触发** | 多端数据同步由用户在桌面客户端手动触发上传，其他端登录后获取 |
| **数据主权** | 用户选择哪些数据上传到云端                  |

#### 4.4.2 客户端同步到服务端的数据

客户端产生的数据中，仅以下数据会同步到服务端：

| 数据项    | 说明                    | 用途                                            |
| ------ | --------------------- | --------------------------------------------- |
| 联系人摘要  | 联系人 ID、平台类型、添加时间      | **重复粉丝检测（重粉功能）** - 通过跨平台联系人识别，判断同一客户是否被多个账号添加 |
| 用户偏好设置 | 翻译偏好、主题选择、快捷键配置（可选同步） | 多端保持设置一致                                      |

> 其他数据（账号信息、订阅/配额、使用统计等）本身由服务端产生和管理，无需客户端同步。

#### 4.4.3 仅保存在本地的数据（不上传）

| 数据项           | 说明                     |
| ------------- | ---------------------- |
| 消息全文          | 各平台的聊天消息内容             |
| 联系人备注字段       | 自定义备注（姓名、年龄、职业、收入、喜好等） |
| 联系人标签         | 彩色标签配置                 |
| 翻译缓存          | 翻译结果本地缓存               |
| 账号 Cookie/登录态 | 敏感凭证，加密本地存储            |
| 自动回复规则        | 触发条件和回复模板              |
| AI 对话历史       | 与 AI 助手的对话上下文          |

#### 4.4.4 多端数据同步流程

`
┌─────────────────┐       手动触发       ┌─────────────────┐
│  桌面客户端 A    │ ──────────────────→ │  后端云存储       │
│  (主设备)        │   上传联系人/设置    │  (临时中转)       │
│                 │                     │                  │

## 6. 非功能需求

### 6.1 性能

| 编号     | 需求            | 指标                                |
| ------ | ------------- | --------------------------------- |
| NFR-01 | 应用启动时间        | 冷启动 ≤ 3 秒，热启动 ≤ 1 秒               |
| NFR-02 | 内存占用          | 基础运行 ≤ 200MB，每个账号 WebView ≤ 150MB |
| NFR-03 | 安装包体积         | Windows 安装包 ≤ 100MB               |
| NFR-04 | 消息列表加载        | 1000 条消息列表 ≤ 500ms                |
| NFR-05 | 翻译响应时间        | ≤ 2 秒（单条文本）                       |
| NFR-06 | AI 首 Token 时间 | ≤ 3 秒                             |
| NFR-07 | 并发账号数         | 同时在线 ≥ 20 个账号                     |
| NFR-08 | CPU 使用率       | 空闲状态 ≤ 5%，活跃状态 ≤ 30%              |

### 6.2 安全

| 编号     | 需求     | 说明                            |
| ------ | ------ | ----------------------------- |
| NFR-09 | 本地数据加密 | SQLite 敏感数据加密存储（Cookie、Token） |
| NFR-10 | 网络安全   | API 通信使用 HTTPS，MQTT 使用 TLS    |
| NFR-11 | 沙箱隔离   | 每个浏览器实例在独立沙箱中运行               |
| NFR-12 | 文件安全   | 自动拦截可执行文件等危险格式传输              |
| NFR-13 | 更新安全   | 更新包签名校验                       |
| NFR-14 | 反调试    | 生产版本禁用开发者工具                   |

### 6.3 隐私

| 编号     | 需求    | 说明                      |
| ------ | ----- | ----------------------- |
| NFR-15 | 本地优先  | 所有消息和联系人优先存储在用户本地，不强制上云 |
| NFR-16 | 数据最小化 | 仅收集必要的运行数据（版本号、崩溃日志）    |
| NFR-17 | 用户控制  | 用户可自主选择是否开启数据上报         |

### 6.4 可用性

| 编号     | 需求      | 说明                |
